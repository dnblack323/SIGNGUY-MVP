# EC6 — Asset Library, Portal, Proofs, Signatures, Approvals

## System overview
EC6 completes the customer-facing document + approval foundation on top of the shared EC1/EC2/EC3/EC4/EC5 platform. Nothing in EC6 duplicates existing systems.

```
Internal Record (Quote/Order/Invoice/WorkOrder)
   → Document (asset-library meta) or File (raw bytes)
   → DocumentLink / DocumentShare (existing EC2)
   → Proof / SignatureRequest (EC6)
   → Public Action Token (single-purpose)  →  Public single-action page
   → Portal Identity (n:1 to Customer) + Magic Link  →  Portal shell (/portal/*)
   → Approval  →  Owning-service transition (audited)  →  EC1/EC2 audit + activity + notifications
```

## Two dependency graphs, zero crossover
Staff: `deps.get_current_user` → `require_permission(Perm.*)`. Rejects any token with `sub_scope="portal"` or `typ="portal_access"`.
Portal: `deps_portal.get_current_portal_identity` → `require_portal_permission("portal:*")`. Reads only tokens whose `sub_scope="portal"` and validates the identity is still `status="active"` on every request.

This is unit-tested from both directions.

## Portal Identity model
- Separate collection `portal_identities`. NOT users.
- Bound to exactly ONE tenant + ONE customer. Unique `(tenant_id, email)`.
- Permission model is backend-authoritative. Bundle presets in `models/portal_identity.py::PRESET_BUNDLES`:
  - `owner_full` — full portal access
  - `billing_only`
  - `approver_only`
  - `viewer_only`
  - `custom` — explicit whitelist from `PORTAL_PERMS`

## Token model
- **Magic link** (login-only): SHA-256 at rest, single-use, audience-scoped to `portal_identity_id`, default 30-minute TTL. Consumed inside `find_and_consume_magic_link()`.
- **Public action** (single-purpose): SHA-256 at rest, revocable, bound to `action + parent_type + parent_id + parent_version?`. Optional `audience_email` for signer binding. Multi-use for `quote_view`/`invoice_view`; single-use for all write actions.

Raw tokens exist for exactly one round-trip: mint → email delivery. They are never persisted or logged.

## Documents / Asset Library
Documents are a metadata layer over the existing `file_records` collection. Each row has `current_file_id` + `version` and version rows in `document_versions`. Category-driven (`artwork`, `proof`, `contract`, `invoice`, `quote`, etc.). `visibility="customer_visible"` is the switch that makes a document eligible for portal listing.

## Proofs
- Parents: `order | order_item | work_order`.
- Each `Proof` has `current_version` + `current_file_id`; every version is an immutable row in `proof_versions`.
- Transitions live in `services/proofs_service.py::ALLOWED_TRANSITIONS`. `blocked`-style semantics come via `changes_requested` + reason. Terminal: `cancelled`, `superseded`, `approved` (implicit final for the version).
- Operational transitions coming from portal or public-token action ALWAYS flow through `transition_proof()` so audit + activity are consistent and the owning module's rules apply.

## Approvals
- Parents: `quote_revision | proof_version | contract | order_item | work_order_summary`.
- Approving/declining is a data-only fact — it does NOT silently mutate pricing/invoice/payment/production. Any operational consequence is performed by the owning service (proofs_service, work_orders_service, quotes_service) explicitly and audited separately.
- Reason required for `request_changes` and `decline`.

## Signatures
- Multi-signer support: `required_signers[]` embedded on the request; each signature ticks the signer's `signed=true` and the request auto-progresses `partially_signed` → `completed`.
- Signature capture is either `typed` (`typed_text`) or `drawn` (`signature_data_ref = file_id`). The composite signed PDF is regenerated by a follow-on hook — the storage slot (`signed_pdf_file_id`) is reserved.

## Portal customer routes
- Every list/detail query filters by `tenant_id AND customer_id` from the JWT. No client trust.
- Portal invoices: excludes drafts.
- Portal documents: `visibility="customer_visible"` AND `archived=false`.
- Portal messages: read from `email_logs` filtered by tenant + customer; provider IDs, error diagnostics, and other internal fields are projected out.
- Portal send-message: subject + body only. Recipient is server-resolved from the tenant record (falls back to owner email). Per-identity rate limit 5/5min. `email_log` written; audit event emitted.

## Public single-action endpoints
- `/api/public/token/introspect` — introspection (no consumption).
- `/api/public/quotes/{id}` + `/api/public/invoices/{id}` — token-scoped read-only.
- `/api/public/proofs/{id}/action` — single-use write. Runs `record_approval` then `transition_proof`.
- `/api/public/signatures/{id}/sign` — single-use write. Runs `record_signature`. Enforces `audience_email` when set.
- `/api/public/quote-request` — unauthenticated public form. Per-IP rate limit. Never writes an approved Quote — creates a `quote_request` row for staff review.
- `/api/public/customer-intake/{id}/submit` — token-scoped. Stores the response raw and computes `staged_changes` for staff review. No silent overwrite of authoritative Customer data.

## Frontend surfaces
- `/portal/*` — customer portal shell, guarded by `PortalAuthContext`. Fully separate from staff `AuthContext`. Mounted OUTSIDE `<AppShell>` in `App.js`.
- `/p/*` — public single-action pages (proof action, quote request). Uses raw axios (no token store).

## Rate limits
- Portal auth (login / magic-link request / magic-link verify): 10/min per IP.
- Portal send-message: 5 / 5min per portal identity.
- Public quote request: 5/min per IP.
